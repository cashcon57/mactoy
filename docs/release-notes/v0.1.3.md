# Mactoy v0.1.3 — The one that actually installs Ventoy

This is the first release that has been **end-to-end verified on real hardware**. Every previous release had a disk-writing bug hiding behind the macOS permission wall — once the permissions path was finally clean, the real bugs surfaced. This release fixes them.

A Ventoy install goes from click to bootable USB in about 45 seconds (30 s of that is the GitHub Releases download, which is out of our control).

## What changed

### Helper architecture

v0.1.2 used `osascript` to elevate the `mactoyd` binary to root, which required an admin password prompt per install and silently needed Full Disk Access granted by hand. v0.1.3 replaces that with a proper **`SMAppService` LaunchDaemon + XPC** flow:

- One-time approval via the native macOS "Background Items Added" notification. No password prompt on subsequent installs.
- The daemon auto-exits after the XPC connection closes, so every install spawns a fresh process — rebuilds and updates take effect immediately, no stale code lingering in memory.
- The daemon verifies the client's Developer ID + bundle identifier before accepting any install plan, closing security issue C2 from the internal audit.
- The app shows a pre-approval **explainer sheet** before triggering macOS's prompt so users know exactly what they're being asked to allow (and that the "background" label is macOS's UI choice, not a functional description — the helper only runs during installs).
- An **"Uninstall helper after this run"** toggle (default on) cleans the system back up after the install completes.

### Full Disk Access flow

macOS TCC does not allow apps to programmatically request Full Disk Access — it has to be granted in Settings by the user. When the helper hits `EPERM` on `/dev/rdisk*`, v0.1.3 surfaces a **numbered-step sheet** with a one-click deep-link into the FDA pane and a "Reveal Mactoy in Finder" button so the user can drag the right bundle into the list. This happens once; after that TCC propagates the grant to `mactoyd` via `AssociatedBundleIdentifiers`.

### Disk-writing path

The heart of the release.

- **`DiskWriter` now releases its raw-disk fd after `fsync`**, before we ask macOS to re-scan the partition table. Previously the class held `/dev/rdisk*` open for the remainder of `VentoyDriver.execute()` (via Swift's deferred `deinit`), and the kernel refuses to re-probe a disk that still has an open writer — so the wait loop was polling a disk its own process was blocking. This was the bug that made every previous release look like it worked but fail at the final format step.
- **`diskutil reloadDisk` is the explicit trigger** we now use to force macOS to re-read the GPT. The previous `diskutil list` approach only queried cached state.
- **Polling tolerates up to 90 s** of DiskArbitration lag before giving up, with `reloadDisk` retried every 10 s, and the `.formatting` phase emits a visible "waiting (Ns / 90s)…" update every 5 s instead of sitting silent.
- **Boot-image layout accepts new-and-old Ventoy releases.** Upstream moved `ventoy.disk.img.xz` from `boot/` to `ventoy/` in a recent release; the driver now checks both paths.

### UI + UX

- **Live download progress.** The 20 MB Ventoy tarball download now ticks a real bytes-done / bytes-total bar instead of an indeterminate sweep.
- **Status bubbles render as rounded rectangles**, not capsules.
- **No more simultaneous "Failed" + progress bar + "Working…" button.** Terminal `.failed` / `.done` progress updates no longer clobber the real failure state set by the XPC reply.
- **Auto-recovery for BTM/launchd desync**: if the Login Items toggle is on but launchd lost its live registration (e.g. after a `launchctl bootout`), the app transparently re-registers and retries the install.

### Security

- XPC peer-signature check in `mactoyd` closes **C2** from the internal audit — any process without a valid Mactoy Developer ID signature is refused a connection, even if it runs as root.
- Ventoy version strings from the GitHub API are re-validated against the same regex applied to user input, so a malformed upstream tag cannot be interpolated into a download URL.

## Verified on real hardware

This is the first release tested against an actual 128 GB USB drive on macOS 26.2. The resulting drive boots and runs Ventoy; the `Ventoy` exFAT partition accepts ISO drops; the `VTOYEFI` FAT32 partition contains the expected boot payload.

## Install

```sh
# Download Mactoy-0.1.3.dmg from the release assets.
open Mactoy-0.1.3.dmg
# Drag Mactoy.app into /Applications.
```

First launch is prompt-free (Apple-notarized). On first Install click, follow the one-time helper approval flow, then the Full Disk Access flow. Neither sheet appears again on subsequent runs.
