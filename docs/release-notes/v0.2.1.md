# Mactoy v0.2.1 — hotfix for v0.2.0 memory regressions + unified logging

v0.2.0 shipped four memory and responsiveness regressions caused by the `@Observable → ObservableObject` migration in the macOS-13.5+/universal port. A user on a MacBook Air M4 reported runaway memory pressure during a flash that froze other apps, followed by repeated crashes after a system restart. **v0.2.0 has been retroactively marked as a prerelease on GitHub** while v0.2.1 ships as the new stable.

This release pins the verifiable causes of that pressure and adds `os.Logger` instrumentation across the launch path, helper lifecycle, install run, and daemon — so future incidents come with a usable diagnostic trail instead of "froze, then crashed."

## What v0.2.0 got wrong (and v0.2.1 fixes)

### 1. Per-update `Task { @MainActor in onUpdate }` spawn during a flash

`ProgressForwarder` (the XPC-facing object that receives progress callbacks from `mactoyd`) used to spawn a fresh `Task { @MainActor in onUpdate(update) }` for every update. On a multi-GB image the daemon emits hundreds of updates per second; v0.2.0 created a Task per update, each one writing to two `@Published` properties (`log.append` + `status = .running`) and re-rendering every `@EnvironmentObject` subscriber.

v0.2.1 throttles to ~30 Hz: phase transitions and terminal `.done` / `.failed` updates are *never* dropped, only intra-phase progress ticks are coalesced. The most-recent throttled-out update is buffered and flushed when the next phase transition or terminal update arrives — so the progress bar always shows 100% before "Install complete" instead of jumping from 95% to done.

### 2. Unbounded `AppState.log` growth

v0.2.0 appended every `ProgressUpdate` to `log: [ProgressUpdate]` indefinitely with no consumer. A long install accumulated thousands of entries (each a phase + message + bytes-done + bytes-total + Date) and pinned them in memory.

v0.2.1 caps `log` at 500 entries via `appendBoundedLog(_:)`. When the cap is hit we drop the oldest 25% in one shot rather than `removeFirst()` per append (amortizes O(n) shifts). The 500 most recent updates is plenty for triage; older entries are not load-bearing for any current view.

### 3. `@Published` thrash from poll loops

Combine's `@Published` fires `objectWillChange` on every assignment — even when the value didn't actually change. `@Observable` (the v0.1.x state layer) compared values automatically and skipped no-op writes; `@Published` does not. v0.2.0's disk-enumeration poll was reassigning identical disk lists every 2 seconds, and the helper-status poll was reassigning identical statuses every 1 second. Each assignment invalidated every view subscribed via `@EnvironmentObject`.

v0.2.1 equality-guards every poll-loop `@Published` write: `applyDiskList`, `setLatestVentoyVersion`, `setAvailableVentoyVersions`, `refreshHelperStatus`, and the helper-status loop in `startHelperPoll`. `DiskTarget` gained an `Equatable` conformance in `MactoyKit` to make the disk-list comparison cheap and exact.

Side effect of the `refreshHelperStatus` guard: the **"Remove the helper after this install"** checkbox no longer resets to its default on every status refresh. Users who manually toggle it now keep their choice across status changes — matches v0.1.x behaviour, which v0.2.0 had unintentionally regressed.

### 4. Force-unwraps in the file picker

`pickFile()` in Flash Image had four `UTType(filenameExtension: "...")!` force-unwraps. If CoreServices's type registry is in an unusual state on the user's machine (rare, but possible after a corrupted system update), one of these returns `nil` and the app traps mid-pick. v0.2.1 uses `compactMap` to silently drop nils and logs a warning if fewer than four lookups succeed.

## Added — `os.Logger` instrumentation

Every Mactoy + daemon binary in v0.2.1 logs to `os.Logger` under subsystem `com.mactoy`. Categories:

- `lifecycle` — app launch banner, first-window setup
- `appstate` — helper status transitions, install `run()` begin / success / failure
- `xpc.progress` — malformed progress drops in the throttle path
- `mactoyd` — daemon startup, accept / reject of XPC clients, executePlan begin / success / failure, connection invalidation
- `ui.picker` — degraded CoreServices state on file pick

`error.localizedDescription` interpolations use `privacy: .private` so home-directory paths are redacted (`<private>`) when a user pastes log output into a public GitHub issue. Disk BSD names, modes, version strings, and helper-status names are `.public` — they're the values most useful for triage.

Capture and attach a log when reporting a bug:

```sh
log show --predicate 'subsystem == "com.mactoy"' --last 1h --info --debug > mactoy-log.txt
```

Documented in the README's new "Reporting bugs" section.
