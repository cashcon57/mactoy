# Mactoy v0.3.0 — Update Ventoy without losing your ISOs

The headline feature: a new **Update Ventoy** mode that updates an existing Ventoy install in-place to a newer Ventoy version, **without erasing the user's ISOs or `/ventoy/` configuration**. Mactoy is the first non-official-Ventoy-team port of this in-place update flow — `Ventoy2Disk.sh` and `Ventoy2Disk.exe` have had it for years; macOS users have been stuck wiping and reinstalling.

## What's new

### Update Ventoy mode

Pick the **Update Ventoy** tab. Mactoy probes the selected disk's partition 2, parses out the currently-installed Ventoy version from `/grub/grub.cfg`, and adapts:

- **Update available** — your disk has Ventoy v1.0.99 and the latest is v1.1.05. One click to update; ISOs preserved.
- **Already up to date** — disk is already on the latest version. The Install Ventoy tab is still there if you want to wipe-and-reinstall.
- **Damaged Ventoy install detected** — partition layout looks like Ventoy but doesn't pass the full validation (e.g. interrupted earlier install). Repair via fresh install (which erases the drive).
- **Not Ventoy** — this disk doesn't have Ventoy on it. Use Install Ventoy.

The version dropdown carries over from Install Ventoy: you can pin a specific Ventoy version to update *to*, or default to "Latest" and resolve at update time. New Ventoy versions automatically work with Mactoy without an app update — the layout has been stable since Ventoy 1.0, and the parser tolerates 4-component / alphanumeric version strings (so `2.0.0-rc1` or `1.2.0.beta1` would Just Work).

### How the update is safe

The on-disk update mirrors `Ventoy2Disk.sh --update` exactly:

1. **Preserve** the Ventoy disk UUID at LBA 0 byte 384 (16 bytes), the 8 reserved sectors at LBA 2040, and your secure-boot toggle (offsets 92 and 17908).
2. **Rewrite** MBR boot code (bytes 0–439), GRUB2 core image at LBA 34 for GPT or LBA 1 for MBR, and the entire 32 MiB VTOYEFI partition.
3. **Restore** the saved UUID, secure-boot bytes, and reserved sectors.

**Partition 1 — where your ISOs and `/ventoy/` config live — is never written.** Not protected by transactional magic; protected by the fact that the update flow simply doesn't issue a write to that partition. If power dies mid-update (~5 seconds total), the bootloader is left half-written and you'll need to re-run the update — but your data is safe regardless.

### Under the hood

- New `VentoyVersionProbe` + `FAT16Reader` in `MactoyKit` — a pure-Swift FAT16 read-only parser plus a Ventoy-specific layer that locates partition 2, validates the layout (P1 starts at sector 2048, P2 size 65536 sectors, FAT16 with `VTOYEFI` label), reads `/grub/grub.cfg`, and extracts `set VENTOY_VERSION="X"`.
- New `probeVentoy` XPC method on the daemon — read-only, safe to call as the user changes their disk selection. App debounces probe RPCs at 300 ms.
- `VentoyDriver.update(plan:progress:)` — byte-level update flow, parallel to the existing fresh-install path. Reuses `DiskWriter`, `VentoyDownloader`, `VentoyBootImages`.
- `VentoyOperation` field on `InstallPlan` (`.freshInstall` | `.updateInPlace`) with backwards-compatible Codable so v0.2.x plans without the field decode as fresh installs.
- 14 new unit tests covering `parseVentoyVersion`, `VentoyProbeResult` round-trip, and `InstallPlan` v1→v2 migration. Build is 30/30 green.

## Known limitations

- **No transactional rollback during update.** Power loss or USB unplug mid-update leaves the bootloader half-written. Recovery: re-run the update. ISOs and config are never at risk.
- **No real-hardware update test.** The fresh-install path has been validated end-to-end on real USB drives (since v0.1.3). The new update path has been validated by static analysis, parser unit tests, and the multi-factor disk-layout sniff that refuses to run on anything that doesn't look like a healthy Ventoy install. Real-world validation comes from user reports — please file an issue with a `log show --predicate 'subsystem == "com.mactoy"' --last 1h` capture if anything misbehaves.
